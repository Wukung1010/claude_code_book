# 第 3 章 环境初始化与首次渲染

> 对应源码主线：src/setup.ts、src/interactiveHelpers.tsx、src/replLauncher.tsx

## 3.1 setup.ts 不是杂项初始化，而是运行环境收口层

很多工程的 setup.ts 都是零散初始化代码，但这个项目不是。

Claude Code 的 setup() 函数承担的是：

1. 建立本次会话的 cwd 和 session 身份
2. 初始化工作目录相关状态
3. 处理 worktree/tmux/teammate 等运行模式
4. 启动 hooks watcher、session memory、后台任务预热
5. 在“首次渲染前必须完成”和“可以延后”之间做边界划分

所以它其实是一个“运行环境收口层”。

## 3.2 setup() 早期逻辑：先把底盘搭起来

setup() 的最前面会做几件非常底层的事：

- 检查 Node 版本
- 处理自定义 sessionId
- 在非 bare 模式下启动 UDS messaging
- 在 swarm 模式下捕获 teammate snapshot
- 恢复 iTerm2 / Terminal.app 可能中断的设置备份

然后会执行一个很关键的动作：

```ts
setCwd(cwd)
captureHooksConfigSnapshot()
initializeFileChangedWatcher(cwd)
```

这三步实际上定义了后续所有文件、hooks、权限、工作目录相关逻辑的根坐标。

尤其是 captureHooksConfigSnapshot()，它体现出一个重要防御思路：

启动时先抓一份 hooks 配置快照，避免运行过程中被悄悄篡改。

把 setup() 早期骨架直接展开，会更容易看清它在“搭底盘”而不是“做业务”：

```ts
if (customSessionId) {
  switchSession(asSessionId(customSessionId))
}

if (!isBareMode() || messagingSocketPath !== undefined) {
  if (feature('UDS_INBOX')) {
    const m = await import('./utils/udsMessaging.js')
    await m.startUdsMessaging(...)
  }
}

if (!isBareMode() && isAgentSwarmsEnabled()) {
  const { captureTeammateModeSnapshot } = await import(...)
  captureTeammateModeSnapshot()
}

setCwd(cwd)
captureHooksConfigSnapshot()
initializeFileChangedWatcher(cwd)
```

这段代码很能说明 setup() 的第一原则：先把 session 身份、消息通道、teammate 现场、cwd、hooks 快照这些运行底座钉住，后面更高层的 worktree、memory、REPL 才有稳定的落脚点。

## 3.3 worktree 路径是 setup() 里最复杂的分支之一

如果启用了 worktree，setup() 会进入一条专门的分支。

这条分支会处理：

1. 是否允许在当前仓库中创建 worktree
2. 是否存在 hook 方式的 worktree create
3. 当前 cwd 是不是已经位于 worktree 内
4. tmux session 是否要一并创建
5. 创建成功后，如何切换 cwd、projectRoot、session worktree state

这一段很关键的代码是：

```ts
worktreeSession = await createWorktreeForSession(
  getSessionId(),
  slug,
  tmuxSessionName,
  worktreePRNumber ? { prNumber: worktreePRNumber } : undefined,
)
```

注意它不是“创建一个临时目录”，而是把 worktree 绑定到 session 语义上。

也就是说，worktree 在这个项目里是会话的一部分，而不是普通 git 辅助功能。

这一点从 worktree 分支后半段也能看得很清楚：

```ts
process.chdir(worktreeSession.worktreePath)
setCwd(worktreeSession.worktreePath)
setOriginalCwd(getCwd())
setProjectRoot(getCwd())
saveWorktreeState(worktreeSession)
clearMemoryFileCaches()
updateHooksConfigSnapshot()
```

这里不是只改了一个 cwd，而是把 projectRoot、worktree state、memory cache、hooks snapshot 一起切换到了新工作树。这说明 setup() 处理的是“完整运行现场迁移”，不是临时跳目录。

## 3.4 setup() 只做首次查询前必须具备的后台注册

源码里有一句非常值得注意的注释：

“Background jobs - only critical registrations that must happen before first query”

对应代码：

```ts
if (!isBareMode()) {
  initSessionMemory()
  if (feature('CONTEXT_COLLAPSE')) {
    require('./services/contextCollapse/index.js').initContextCollapse()
  }
}
void lockCurrentVersion()
```

这说明 setup() 并不试图把所有后台服务都拉起来，而是只启动“首次 query 之前必须存在”的组件。

这是一个很成熟的启动设计：

- 首屏之前做最小必要工作
- 不要把所有“未来可能有用”的优化都堵在第一帧前面

也就是说，setup() 在首次 query 前只注册“缺了就没法正常跑”的后台能力，而不是把所有后台系统一次性全拉起来。这和第 2 章里看到的冷启动分层是一致的。

## 3.5 renderAndRun 为什么单独抽出来

interactiveHelpers.tsx 里的 renderAndRun 很短，但很关键：

```tsx
export async function renderAndRun(root: Root, element: React.ReactNode): Promise<void> {
  root.render(element)
  startDeferredPrefetches()
  await root.waitUntilExit()
  await gracefulShutdown(0)
}
```

这个函数把交互程序的“主生命周期尾板”固定住了：

1. 把 UI 渲染出来
2. 首屏之后再启动延迟预取
3. 等待 UI 退出
4. 做优雅关停

这里的重点不是代码量，而是边界非常清楚。

## 3.6 为什么要把 prefetch 延迟到首屏之后

main.tsx 里定义的 startDeferredPrefetches()，本质上是在说：

“这些事情很有价值，但不该阻塞第一帧。”

延迟启动的内容包括：

- initUser()
- getUserContext()
- prefetchSystemContextIfSafe()
- getRelevantTips()
- Bedrock / Vertex 认证预取
- 文件数量统计
- Analytics gate 初始化
- MCP registry 预取
- model capabilities 刷新
- settings/skills change detector 初始化

这说明整个工程对启动性能是有明显意识的。

换句话说，这不是“把逻辑塞到 setTimeout 里”，而是显式地区分：

- 首次交互必须的逻辑
- 用户正在输入第一条消息时可以悄悄完成的逻辑

## 3.7 replLauncher.tsx 的职责：隔离 App 与 REPL 的装配

replLauncher.tsx 代码很短：

```tsx
export async function launchRepl(
  root: Root,
  appProps: AppWrapperProps,
  replProps: REPLProps,
  renderAndRun: (root: Root, element: React.ReactNode) => Promise<void>,
): Promise<void> {
  const { App } = await import('./components/App.js')
  const { REPL } = await import('./screens/REPL.js')
  await renderAndRun(
    root,
    <App {...appProps}>
      <REPL {...replProps} />
    </App>,
  )
}
```

它的意义不在“封装了一层 JSX”，而在于：

1. 把 App 级 Provider 和 REPL 主屏幕拆开
2. 通过动态 import 延后较重的 UI 模块加载
3. 统一所有 main.tsx 中进入 REPL 的路径

所以 launchRepl 是“REPL 装配点”，而不是“REPL 本身”。

## 3.8 showSetupScreens：首次交互前的人机边界

interactiveHelpers.tsx 里另一个很重要的函数是 showSetupScreens()。

这里负责的并不只是 onboarding，而是：

- onboarding
- trust dialog
- GrowthBook 在 trust 之后重新初始化
- system context 在 trust 后才允许预取
- mcp.json server approvals
- external includes warning
- 自定义 API key 审批
- bypass permissions 提示
- auto mode opt-in

这个函数说明 Claude Code 的安全边界不是“工具执行时才检查”，而是在进入交互态前就先建立工作区信任边界。

## 3.9 这一章的阅读结论

setup.ts、interactiveHelpers.tsx、replLauncher.tsx 三个文件合起来，回答的是一个问题：

“命令解析结束之后，程序如何进入一个可交互、可退出、可延迟预热、带有工作区安全边界的 REPL 运行态？”

总结成一句话：

- setup.ts 负责搭环境底盘
- showSetupScreens 负责建立交互前的人机信任边界
- renderAndRun 负责托管 UI 生命周期
- replLauncher.tsx 负责把 App 和 REPL 真正装进 Ink 根节点

下一章开始看 context.ts 和 system prompt 相关设计，因为 Agent 真正“知道什么”，取决于上下文系统而不是 UI。

## 3.10 setup() 的函数级执行链

如果把 setup() 从头到尾拉直，它其实很像一个“环境安装流水线”：

1. 校验 Node 版本
2. 处理 custom session id
3. 启动 UDS messaging 与 teammate snapshot
4. 恢复可能中断的终端设置备份
5. setCwd(cwd)
6. captureHooksConfigSnapshot()
7. initializeFileChangedWatcher(cwd)
8. 如有需要，创建并切换 worktree / tmux
9. 初始化 session memory、context collapse、version lock 等后台前置能力
10. 启动后续预取与缓存预热

这条链路很重要，因为它说明 setup() 不是简单初始化，而是在建立“本会话的实际运行现场”。

## 3.11 为什么 setup() 和 getCommands() 可以并行

在 main.tsx 中，setup() 会和 getCommands()/getAgentDefinitionsWithOverrides() 尝试并行。

这背后依赖一个假设：

- 非 worktree 情况下，setup() 不会改变最终 cwd
- commands/agents 装载主要是文件读取与缓存，不依赖 setup() 先完成

但如果启用了 worktree，这个假设不再成立，因为 setup() 里可能 process.chdir(worktreePath)。

因此 main.tsx 才会在 worktreeEnabled 时关闭这段并行优化。

这正是大型 CLI 启动优化里很典型的一类判断：

- 能并行的尽量并行
- 但不能为了并行破坏语义正确性

## 3.12 showSetupScreens() 的真正角色：交互式安全边界闸门

showSetupScreens() 不是“进 REPL 前弹几个窗口”，而是交互式模式的安全闸门。

它会串起下面这些动作：

1. 必要时先做 onboarding
2. 检查 trust dialog，未信任则阻塞在这里
3. trust 成立后，setSessionTrustAccepted(true)
4. resetGrowthBook() 并重新 initializeGrowthBook()
5. 这时才允许 prefetchSystemContext
6. 检查 mcp.json approvals 与 external includes warning
7. 再继续执行 API key、bypass permissions、auto mode opt-in 等交互式提示

换句话说，showSetupScreens() 不是 UX 附件，而是“在 REPL 真正运行前，先把工作区信任边界钉死”。

## 3.13 renderAndRun() 为什么重要到值得单独抽成函数

renderAndRun() 很短，但它把 REPL 生命周期的尾板完全固定住了：

```tsx
root.render(element)
startDeferredPrefetches()
await root.waitUntilExit()
await gracefulShutdown(0)
```

这意味着进入交互态之后，宿主程序的统一约束是：

1. 先 render
2. 再做首屏后预热
3. 最后必须走 graceful shutdown

这类统一尾板的价值非常大，因为否则每条进入 REPL 的路径都可能自己复制一遍渲染与收尾逻辑，最终变成多个不一致的退出行为。

## 3.14 launchRepl() 的真正作用：推迟 UI 主模块求值

replLauncher.tsx 中最容易被忽略的一点，是它用的是动态 import：

```ts
const { App } = await import('./components/App.js')
const { REPL } = await import('./screens/REPL.js')
```

这不仅仅是“代码拆分”，还意味着：

- 在命令解析和 setup 阶段，不急着求值庞大的 UI 树
- 把更重的 React/Ink 组件装载推迟到真正需要进入 REPL 时

对 CLI 工程来说，这会直接影响首屏可感知延迟。

## 3.15 startDeferredPrefetches() 的设计边界

main.tsx 里的 startDeferredPrefetches() 很值得和 setup() 对照着看。

两者的关系是：

- setup() 只做 first query 之前必须具备的能力
- startDeferredPrefetches() 做用户输入第一条消息时可以悄悄准备的能力

这两层分开之后，启动路径就有了非常清晰的性能边界：

### setup 前置区

必须在首个交互前完成。

### deferred prefetch 区

可以和用户观察首屏、思考第一条输入的时间重叠。

这是这套工程启动性能控制里最成熟的地方之一。

## 3.16 这一章和后续章节怎么衔接

第 3 章把“入口装配”进一步落成了“实际运行现场”。

1. 它承接第 2 章，因为前一章只解决命令和模式如何被装起来，这一章才真正把 cwd、worktree、hooks watcher、trust dialog 和 Ink 生命周期落到地面。
2. 它会直接通向第 4 章，因为 REPL 真正开始运行之后，下一层关键问题就不再是 UI 本身，而是 system context、user context 和 system prompt 怎样被准备出来。
3. 它也会回流到第 10 章和第 14 章，因为这里建立的 REPL 壳、首屏预热和交互前信任边界，后面都会继续影响输入桥接和 remote/worktree 执行拓扑。

所以第 3 章的定位，不只是“初始化章节”，而是统一运行时从装配态进入可交互运行态的真正落点。
