# Codebase Understanding

## Overview

系统化理解代码仓库的框架。通过结构化的维度分析，快速掌握项目全貌，为后续开发、调试或重构提供扎实的上下文基础。

**核心原则：理解优先于行动，探索先于实现。**

## When to Use

**执行方式：使用 `Explore` subagent 执行**
代码仓库分析是独立的原子操作，不依赖主对话上下文，应使用 subagent 隔离执行，避免占用主会话资源。

适用场景：
- 首次接触项目，需要全局认知
- 接手遗留代码，需要快速理解业务逻辑
- 进行重大重构前，需要评估影响范围
- 调试复杂问题，需要理解数据流和调用链
- 代码审查前，需要掌握项目结构

不适用场景：
- 单个文件/函数的简单理解（直接用 Read/Grep 即可）
- 已明确知道需要修改的具体位置
- 需要与当前代码库状态对比的增量分析

## 执行模式

### 单项目模式（项目数 ≤ 5 个模块）
由 **1 个 Explore subagent** 完成全部分析任务，主会话负责最终验证。

### 多项目模式（项目数 > 5 个模块）
由 **主会话 + 多个 Explore subagent** 并行执行：
- 主会话：Phase 1 全局扫描 + Phase 4 报告验证
- subagent：Phase 2/3 模块深度分析

### 并发控制
- 每次最多启动 **3 个** Explore subagent
- 每个 subagent 承担 **连续的任务块**（如模块 A-C），避免任务碎片化
- 主会话维护 TASK_QUEUE.md 追踪执行状态

## 任务分发机制

**任务状态流转：**
```
pending → in_progress → completed
                    ↘ failed（重置为 pending，可重试 2 次）
```

**任务列表文件：**
```
docs/codebase-understanding/[monorepo]/.tasks/
└── TASK_QUEUE.md   # 任务队列
```

**TASK_QUEUE.md 模板：**
```markdown
# 任务队列

## 待执行
- [ ] [模块] [文件] - 描述

## 执行中
- [>] [模块] [文件] - 描述 (subagent: agent-1)

## 已完成
- [x] [模块] [文件] - 描述
```

**subagent 执行流程：**
1. 读取 TASK_QUEUE.md
2. 选择一个 pending 任务
3. 更新状态为 `in_progress` + 自己的 agent ID（使用 Edit 工具）
4. 执行分析任务
5. 生成所有指定的产出文件
6. 验证所有文件已创建
7. 更新状态为 `completed`（使用 Edit 工具）

## subagent prompt 模板

```
# 任务：分析 [模块/文件]

## 目标
[具体要分析什么]

## 产出（必须全部完成）
1. 主报告：写入 [具体路径]
2. 文件级报告（必须全部）：
   - [文件1路径]
   - [文件2路径]
   - [文件3路径]
3. 更新 TASK_QUEUE.md：使用 Edit 工具将任务状态从 in_progress 改为 completed

## 验证产出
完成前必须确认以下文件存在：
- [列出预期生成的所有文件]

## 约束
- 不修改任何代码
- 必须使用 Edit 工具更新 TASK_QUEUE.md
- 必须在产出中包含验证步骤

## 上下文
- 项目根目录：[实际路径]
- monorepo 名称：[名称]
```

## 执行流程

### Phase 1: 快速全局扫描

**目标：建立高层视图**

```
Step 1: 获取项目上下文
├── 读取 README.md（项目目的、架构概述）
├── 读取 package.json（依赖、脚本、入口配置）
└── 观察顶层目录结构（识别 monorepo / 单体）

Step 2: 识别入口点
├── 查找 main/index/App 等入口文件
├── 识别 CLI/API/页面 入口类型
└── 确定分析起点

Step 3: 建立高层视图
├── 画出模块划分草图
├── 识别共享代码位置（utils/common/shared）
└── 标记需要深度分析的核心模块

完成标志：能画出项目的模块划分草图
```

### Phase 2: 聚焦核心模块深度分析

**目标：深入理解核心业务逻辑**

```
Step 4: 追踪关键调用链
├── 从入口点开始
├── 追踪核心业务逻辑路径
└── 记录关键函数和数据转换点

Step 5: 分析核心模块
├── 深度阅读核心模块代码
├── 提取关键类型/接口定义
├── 理解模块职责和对外接口
└── 分析模块间依赖关系

Step 6: 理解数据流
├── 识别核心数据模型
├── 追踪数据如何在模块间传递
└── 理解状态管理方式

完成标志：能详细描述核心模块的职责和交互方式
```

### Phase 3: 迭代验证与整合

**目标：交叉验证，提炼关键发现**

```
Step 7: 交叉验证
├── 用找到的局部理解验证全局视图
├── 修正之前理解的偏差
├── 补充遗漏的关键联系
└── 读测试文件辅助理解（可选，有则读，无则跳）

Step 8: 提炼输出
├── 整理核心发现
├── 输出关键结论（不是完整文档）
└── 标注仍存在的疑问点

完成标志：能回答最初的探索问题

### Phase 4: 报告验证

**主会话执行此阶段，验证报告完整性和正确性**

**目标：确保报告准确性和完整性**

```
Step 9: 验证报告完整性
├── 检查总报告 5 个文件是否都存在
├── 检查每个模块的 SUMMARY.md 是否存在
├── 检查每个模块的 files/ 目录是否非空
└── 确认文件级报告数量符合预期

Step 10: 交叉验证关键结论
├── 从文件级报告追溯到 SUMMARY.md 结论
├── 验证数据流描述与实际代码是否匹配
└── 标记不确定的内容为"待验证"

Step 11: 补充缺失信息
├── 识别报告中的空白
├── 补充关键发现
└── 标注仍存在的疑问点
```

**完成标志：**
- 所有计划文件存在
- 关键结论有文件级证据支撑
- 标记的疑问点不影响核心理解

**报告验证清单：**

| 验证项 | 要求 | 验证方法 |
|--------|------|----------|
| 总报告文件 | 5 个 | `ls` 检查根目录 |
| 模块 SUMMARY.md | 每个模块 1 个 | `find` 计数 |
| 文件级报告 | 每个模块至少 3 个 | `find` 计数 |
| SUMMARY.md 内容 | 包含"一句话描述"和"快速结论"(≥5条) | `grep` 检查 |
| KEYFINDINGS.md | 至少 8 条发现 | `grep -c` 计数 |
| 证据追溯 | 每条 KEYFINDINGS 有 `[来源: 文件]` 标记 | `grep` 检查 |

**验证命令：**
```bash
# 验证总报告文件
ls docs/codebase-understanding/[monorepo]/*.md | wc -l  # 应为 5

# 验证模块数量
find docs/codebase-understanding/[monorepo]/artifacts/modules -maxdepth 1 -type d | wc -l

# 验证文件级报告
find docs/codebase-understanding/[monorepo]/artifacts/modules/*/files -name "*.md" | wc -l

# 验证快速结论数量
grep -c "^- " docs/codebase-understanding/[monorepo]/SUMMARY.md  # 应 ≥5
```

## Explore subagent 执行指南

### 何时使用 Explore agent
- 模块数量 > 5 时，主会话只做 Phase 1 + Phase 4
- Phase 2/3 的模块分析交给 Explore agent 并行执行

### Explore agent prompt 要点
```
必须包含：
1. 任务目标（具体分析哪个模块）
2. 产出路径（精确的文件路径）
3. 验证步骤（确认文件存在）
4. 约束（不修改代码、只读）
5. 来源标记要求（每条发现要标注来源）

禁止：
- 不要说"探索代码库"这种模糊指令
- 不要遗漏具体的产出文件路径
```

### 任务分配建议
| 模块数 | 执行方式 | 说明 |
|--------|----------|------|
| ≤ 5 | 1 个 Explore agent | 全部分析 |
| 6-10 | 2 个 Explore agent | 每人 3-5 模块 |
| > 10 | 3 个 Explore agent | 每人 3-4 模块 + 主会话处理剩余 |

### 验证失败处理
1. 文件数量不足 → subagent 补充
2. 内容深度不够 → subagent 补充
3. 证据缺失 → subagent 补充特定文件分析
4. 重试 2 次仍失败 → 标记为 failed，主会话接管

```bash
# 了解项目规模
find . -type f \( -name "*.ts" -o -name "*.tsx" \) | wc -l
find . -type f -name "*.test.ts" | wc -l

# 查找入口文件
ls -la src/index.* src/main.* src/App.*
cat package.json | grep -A2 '"main"'

# 分析依赖
cat package.json | grep -E '"@' | head -20
```

## Common Mistakes

| 错误 | 问题 | 正确做法 |
|------|------|----------|
| 直接写代码 | 缺乏全局理解，后续返工 | 先理解再动手 |
| 只看表面结构 | 忽略深层设计意图 | 追踪关键调用链 |
| 假设命名即语义 | 可能是历史遗留命名 | 结合业务场景理解 |
| 忽视测试代码 | 测试反映使用方式 | 阅读关键测试 |
| 报告深度不足 | 只写摘要，缺乏代码证据 | 每个结论标注来源文件和行号 |
| 文件级报告不足 | 每个模块只写 1-2 个文件 | 每个模块至少 3 个文件级报告 |
| 缺少证据追溯 | KEYFINDINGS 无法验证 | 每条发现标注 `> 来源: 文件:行号` |
| 跳过验证 | 报告不完整就结束 | 必须执行 Phase 4 验证清单 |

## Tips

- **从 README 开始**：文档往往反映项目的核心价值
- **追踪入口点**：找到 main/run/start 文件，理解启动流程
- **识别共享代码**：utils/, shared/, common/ 是理解全局的关键
- **关注 type 定义**：TypeScript 中类型往往揭示数据模型
- **善用 LSP**：用 go-to-definition 快速定位实现

## 约束

- **不修改任何代码** - 仅用于理解和分析
- **不执行测试** - 只读测试文件辅助理解，不运行测试
- **报告输出目录** - `docs/codebase-understanding/[monorepo-name]/`
- **优先读而非搜** - 使用 Read 工具直接读取文件
